Personal Relevance Digest

Project Overview

A self-hosted n8n workflow that combines three personally relevant data sources such as GenSan rain forecasts, nationwide earthquake activity, and Philippine national news, into a single AI-generated digest email, sent only when something new has actually happened. The project combines API integration, web scraping, database-backed deduplication, and AI synthesis into one production pipeline.

Business Problem

Staying informed about daily risks and news in the Philippines means checking multiple disconnected sources like a weather service, an earthquake bulletin, a news site, none of which are personalized or consolidated. Manually checking each one is easy to forget, and most of what's on each site on any given day isn't actually new, making it hard to tell what's worth attention versus what's already been seen.

Solution

I built a scheduled n8n workflow with three independent branches, Weather (Open-Meteo API), Earthquakes (PHIVOLCS bulletin, scraped), and News (ABS-CBN, parsed from the site's embedded page data), each filtering to what's relevant and checking a Postgres database to determine what's genuinely new since the last run. The three branches merge into a single gate: if nothing new was found, the workflow ends silently; if something was found, the combined data is handed to Google Gemini, which writes a single readable HTML digest sent via Gmail. The workflow runs three times daily.

Key Features
  • Three independent, parallel data branches merged into one pipeline
  • Postgres-backed deduplication (Neon) so the same earthquake or article never triggers a repeat alert
  • "Only email if something's new" gate or no noise on quiet days
  • AI-synthesized digest (Gemini) that writes one coherent summary across all three sources, rather than three separate lists
  • Direct extraction of a site's embedded JSON data for the JavaScript-rendered news source, bypassing traditional CSS-selector scraping
  • Fully HTML-formatted email output with clickable headlines and summaries
  • Time-zone-correct "is this today" logic, handled differently per source depending on the timestamp format each one provides
Technologies Used
  • n8n (self-hosted on Hostinger)
  • Open-Meteo API
  • ScrapingBee (for JavaScript-protected scraping)
  • Neon (serverless Postgres)
  • Google Gemini API (3.5 Flash-Lite)
  • Gmail API
  • JavaScript (Code nodes) and Luxon for timezone handling
Challenges or Learnings

Two of my three originally planned data sources sat behind bot protection that wasn't worth the cost or fragility to fight, so I made the call to swap both for sources that were either a clean public API or a plain HTTP-accessible page without changing the pipeline's overall architecture.

I also ran into an n8n behavior that isn't obvious until you hit it: a node that returns zero output items causes n8n to skip every node downstream of it, which is a serious problem when "zero results" (no rain today, no new earthquakes) is the normal, expected outcome rather than an edge case. I had to explicitly handle this in three separate places in the pipeline, learning to distinguish "always output a placeholder so the chain keeps moving" from "filter that placeholder back out before it reaches something that shouldn't see it," since carelessly letting a placeholder reach a database insert step caused a separate downstream error.

Timezones were a recurring theme, one source had no true machine-readable date, requiring a string-matching workaround, while another needed explicit UTC conversion. The same "is this today" question needed a different technical answer for each source. I also hit deprecated-model errors twice from Gemini's lineup changing mid-build, a reminder that AI model availability isn't a "set once" decision the way a REST API endpoint usually is.

Outcome or Impact

The workflow runs three times daily, fully unattended, correctly detecting new rain risk, new earthquakes, and new headlines while staying silent on days with nothing to report. What started with a single planned data source became a working case study in adapting scope around real-world constraints such as anti-bot protection, API deprecations, and a finite testing budget.

Beyond the technical build, the digest has become part of my daily routine: knowing rain probability in advance helps me decide whether to cover my motorcycle and plan my day around it, the earthquake updates keep me aware of activity across the country, and having curated headlines and summary land in my inbox makes me more likely to actually read the news instead of skipping it.
